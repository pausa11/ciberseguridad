\section{Introducción}

La ciberseguridad ofensiva es una disciplina que permite a los profesionales de tecnología comprender y anticipar las técnicas utilizadas por actores maliciosos, con el fin de fortalecer la postura de seguridad de las organizaciones. En el marco de esta actividad académica, se exploran cinco vectores de ataque sobre la máquina virtual Metasploitable2, un entorno diseñado intencionalmente con vulnerabilidades conocidas para fines educativos.

Las tareas desarrolladas abarcan la explotación de backdoors mediante el framework Metasploit, ataques de inyección SQL y ejecución de comandos sobre la aplicación web DVWA (\textit{Damn Vulnerable Web Application}), ataques de fuerza bruta con Burp Suite e Hydra, y finalmente el cracking de contraseñas fuera de línea con John the Ripper.

Todas las pruebas se realizaron en un entorno aislado de red (modo Solo Anfitrión) sobre una Mac Mini M4, utilizando UTM como hipervisor para gestionar las máquinas virtuales Kali Linux (atacante) y Metasploitable2 (víctima).

\section{Tarea 1: Explotación de Backdoor en vsftpd 2.3.4}

\subsection{Descripción de la vulnerabilidad}

La vulnerabilidad CVE-2011-2523 afecta al demonio FTP \texttt{vsftpd} en su versión 2.3.4. Esta versión fue comprometida en su repositorio de distribución oficial en julio de 2011, cuando un atacante introdujo un backdoor en el código fuente. El mecanismo del backdoor consiste en que, al recibir una cadena de usuario que contiene los caracteres \texttt{:)} (carita feliz), el proceso abre una shell interactiva en el puerto TCP 6200 con privilegios de root, sin requerir autenticación alguna.

Esta vulnerabilidad es considerada crítica ya que otorga acceso completo al sistema con el máximo nivel de privilegios de forma trivial.

\subsection{Herramientas utilizadas}

\begin{itemize}
  \item \textbf{Nmap}: Escáner de red utilizado para descubrir hosts y servicios activos en la red, identificando la versión del servicio FTP vulnerable.
  \item \textbf{Metasploit Framework}: Framework de explotación de código abierto. Provee el módulo \texttt{exploit/unix/ftp/vsftpd\_234\_backdoor} que automatiza el proceso de activación y conexión al backdoor.
\end{itemize}

\subsection{Pasos realizados}

\subsubsection{Reconocimiento — Escaneo de red con Nmap}

El primer paso consistió en descubrir los hosts activos en la red y enumerar los servicios expuestos por Metasploitable2. Se ejecutó el siguiente comando desde Kali Linux:

\begin{lstlisting}[language=bash, caption={Escaneo de servicios con Nmap}]
nmap -sV 192.168.128.0/24
\end{lstlisting}

El resultado confirmó que Metasploitable2 (192.168.128.2) expone 22 servicios TCP, entre ellos el servicio FTP corriendo \texttt{vsftpd 2.3.4} en el puerto 21.

\begin{figure}[H]
  \centering
  \includegraphics[width=0.92\textwidth]{../img/tarea1/t1_nmap_scan_kali.png}
  \caption{\textit{Resultado del escaneo Nmap mostrando los servicios activos en Metasploitable2, incluyendo vsftpd 2.3.4 en el puerto 21.}}
\end{figure}

\begin{figure}[H]
  \centering
  \includegraphics[width=0.92\textwidth]{../img/tarea1/t1_nmap_scan_port21_kali.png}
  \caption{\textit{Detalle del servicio FTP vsftpd 2.3.4 identificado en el puerto 21 de Metasploitable2.}}
\end{figure}

\subsubsection{Explotación con Metasploit Framework}

Se inició \texttt{msfconsole} y se cargó el módulo específico para la vulnerabilidad CVE-2011-2523:

\begin{lstlisting}[language=bash, caption={Configuración del módulo de explotación en Metasploit}]
use exploit/unix/ftp/vsftpd_234_backdoor
set RHOSTS 192.168.128.2
run
\end{lstlisting}

El módulo establece una conexión al servicio FTP de la víctima, envía las credenciales que contienen los caracteres \texttt{:)} para activar el backdoor, y luego se conecta al puerto 6200 donde la shell queda a la escucha. Al ejecutar el exploit, Metasploit confirmó la apertura de una sesión interactiva con el mensaje:

\begin{lstlisting}
[+] 192.168.128.2:21 - UID: uid=0(root) gid=0(root)
[*] Found shell.
[*] Command shell session 1 opened (192.168.128.3:44327 -> 192.168.128.2:6200)
\end{lstlisting}

\begin{figure}[H]
  \centering
  \includegraphics[width=0.92\textwidth]{../img/tarea1/t1_msfconsole_exploit_unix_ftp_vsftpd_234_backdoor.png}
  \caption{\textit{Ejecución del exploit en Metasploit Framework confirmando acceso root mediante backdoor en vsftpd 2.3.4.}}
\end{figure}

\subsection{Funcionamiento del ataque}

El ataque sigue el siguiente flujo:

\begin{enumerate}
  \item \textbf{Reconocimiento}: Nmap identifica el servicio vsftpd 2.3.4 en el puerto 21.
  \item \textbf{Activación del backdoor}: El módulo de Metasploit se conecta al puerto 21 y envía un nombre de usuario que termina con \texttt{:)} (por ejemplo, \texttt{user:)}). Esto activa el código malicioso insertado en el binario.
  \item \textbf{Apertura de la shell}: El proceso vsftpd comprometido abre una shell root en el puerto TCP 6200.
  \item \textbf{Conexión}: Metasploit se conecta automáticamente al puerto 6200 y establece una sesión interactiva con privilegios de root (\texttt{uid=0}).
\end{enumerate}

\subsection{Resultado}

Se obtuvo acceso total a Metasploitable2 con privilegios de superusuario (\texttt{uid=0(root) gid=0(root)}), demostrando el riesgo crítico que representa ejecutar software obtenido de fuentes comprometidas sin verificar su integridad.

\section{Tarea 2: SQL Injection en DVWA}

\subsection{Descripción de la vulnerabilidad}

La inyección SQL (\textit{SQL Injection}) es una vulnerabilidad de seguridad que permite a un atacante interferir con las consultas que una aplicación web realiza a su base de datos. Se produce cuando la entrada del usuario se incorpora directamente en una consulta SQL sin el debido saneamiento, permitiendo manipular la lógica de la consulta.

En DVWA con nivel de seguridad \textit{Low}, el parámetro \texttt{id} del módulo SQL Injection se concatena directamente en la consulta SQL sin ningún tipo de validación ni uso de consultas parametrizadas, lo que lo hace completamente vulnerable a este tipo de ataque.

\subsection{Herramientas utilizadas}

\begin{itemize}
  \item \textbf{DVWA} (\textit{Damn Vulnerable Web Application}): Aplicación web deliberadamente vulnerable que corre sobre el servidor Apache de Metasploitable2. Provee el módulo SQL Injection con distintos niveles de dificultad configurables.
  \item \textbf{sqlmap}: Herramienta de código abierto que automatiza la detección y explotación de vulnerabilidades de inyección SQL. Soporta múltiples técnicas de inyección (boolean-based blind, error-based, time-based blind, UNION query) y permite enumerar bases de datos, tablas, columnas y volcar su contenido.
\end{itemize}

\subsection{Pasos realizados}

\subsubsection{Configuración del nivel de seguridad en DVWA}

Se accedió a DVWA en \texttt{http://192.168.128.2/dvwa} con las credenciales \texttt{admin} / \texttt{password}. En la sección \textbf{DVWA Security} se configuró el nivel en \textbf{Low} para exponer la vulnerabilidad sin mitigaciones.

\subsection{Funcionamiento del ataque}

El ataque de inyección SQL sigue el siguiente flujo:

\begin{enumerate}
  \item \textbf{Identificación del punto de inyección}: sqlmap envía payloads de prueba al parámetro \texttt{id} de la URL y analiza las respuestas para detectar comportamientos anómalos que delaten una vulnerabilidad SQL.
  \item \textbf{Confirmación de la técnica}: Se identificaron cuatro técnicas de inyección aplicables: boolean-based blind, error-based (FLOOR), time-based blind (SLEEP) y UNION query de 2 columnas.
  \item \textbf{Enumeración}: Aprovechando la técnica UNION query, sqlmap extrae los nombres de bases de datos, tablas y columnas del servidor MySQL directamente desde las tablas del sistema (\texttt{information\_schema}).
  \item \textbf{Exfiltración de datos}: Se vuelca el contenido de la tabla \texttt{users}, obteniendo nombres de usuario, hashes MD5 de contraseñas y otros campos.
  \item \textbf{Cracking de hashes}: sqlmap identifica automáticamente el algoritmo MD5 y aplica un ataque de diccionario, recuperando las contraseñas en texto plano.
\end{enumerate}

\section{Tarea 3: Command Execution en DVWA}

\subsection{Descripción de la vulnerabilidad}

La vulnerabilidad de ejecución de comandos (\textit{Command Injection}) se produce cuando una aplicación web incorpora datos proporcionados por el usuario como parte de un comando del sistema operativo sin aplicar ningún tipo de sanitización. En PHP, funciones como \texttt{shell\_exec()}, \texttt{exec()} o \texttt{passthru()} ejecutan el comando construido directamente en el sistema operativo subyacente. Si el valor de entrada no es validado, un atacante puede anexar comandos arbitrarios utilizando operadores de encadenamiento de shell como \texttt{;}, \texttt{\&\&} o \texttt{|}.

En DVWA con nivel de seguridad \textit{Low}, el módulo \textbf{Command Execution} recibe un campo de dirección IP para ejecutar un ping y concatena la entrada directamente en la llamada al sistema sin ningún filtrado, lo que permite inyectar comandos adicionales con pleno acceso a los recursos del servidor.

\subsection{Herramientas utilizadas}

\begin{itemize}
  \item \textbf{DVWA} (\textit{Damn Vulnerable Web Application}): Aplicación web vulnerable que corre en el servidor Apache de Metasploitable2. El módulo \textit{Command Execution} expone la vulnerabilidad de inyección de comandos en nivel de seguridad Low.
  \item \textbf{Navegador web (Firefox en Kali Linux)}: Utilizado para interactuar con la interfaz de DVWA y enviar los payloads de inyección de comandos.
\end{itemize}

\subsection{Pasos realizados}

\subsubsection{Acceso al módulo Command Execution en DVWA}

Desde el navegador en Kali Linux se accedió a DVWA en \texttt{http://192.168.128.2/dvwa} con las credenciales \texttt{admin} / \texttt{password}, con el nivel de seguridad establecido en \textbf{Low}. Se navegó a la sección \textbf{Command Execution} del menú lateral.

\begin{figure}[H]
  \centering
  \includegraphics[width=0.92\textwidth]{../img/tarea3/t3_command_execution_kali.png}
  \caption{\textit{Módulo Command Execution de DVWA con el formulario de ping listo para recibir entrada.}}
\end{figure}

\subsubsection{Inyección de comando para obtener /etc/passwd}

En el campo de dirección IP se ingresó el siguiente payload, que encadena el comando \texttt{cat /etc/passwd} al ping legítimo usando el operador \texttt{;}:

\begin{lstlisting}[language=bash, caption={Payload de inyección para leer /etc/passwd}]
127.0.0.1; cat /etc/passwd
\end{lstlisting}

Al enviar el formulario, el servidor ejecutó primero el ping a \texttt{127.0.0.1} y luego el comando \texttt{cat /etc/passwd}, mostrando el contenido completo del archivo en la página de respuesta.

\begin{figure}[H]
  \centering
  \includegraphics[width=0.92\textwidth]{../img/tarea3/t3_command_execution_cat_passwd.png}
  \caption{\textit{Resultado de la inyección mostrando el contenido completo de /etc/passwd del servidor Metasploitable2.}}
\end{figure}

El archivo \texttt{/etc/passwd} obtenido lista 19 usuarios del sistema, entre ellos cuentas con shell interactiva como \texttt{root}, \texttt{msfadmin}, \texttt{user}, \texttt{service} y \texttt{postgres}.

\subsubsection{Inyección de comando para obtener /etc/shadow}

De manera análoga, se inyectó el comando para leer el archivo \texttt{/etc/shadow}, que contiene los hashes de las contraseñas del sistema:

\begin{lstlisting}[language=bash, caption={Payload de inyección para leer /etc/shadow}]
127.0.0.1; cat /etc/shadow
\end{lstlisting}

\begin{figure}[H]
  \centering
  \includegraphics[width=0.92\textwidth]{../img/tarea3/t3_command_execution_cat_shadow.png}
  \caption{\textit{Resultado de la inyección mostrando el contenido de /etc/shadow con los hashes de contraseñas del servidor.}}
\end{figure}

En condiciones normales, el archivo \texttt{/etc/shadow} solo es legible por el usuario \texttt{root}. El hecho de que el servidor web lo exponga evidencia que el proceso PHP se ejecuta con privilegios elevados en Metasploitable2.

\subsection{Funcionamiento del ataque}

El ataque de inyección de comandos sigue el siguiente flujo:

\begin{enumerate}
  \item \textbf{Identificación del punto de inyección}: El módulo Command Execution de DVWA expone un campo de texto donde el usuario ingresa una dirección IP para ejecutar un ping. Internamente, el código PHP construye el comando de la forma: \texttt{shell\_exec("ping -c 3 " . \$\_REQUEST['ip'])}.
  \item \textbf{Construcción del payload}: El atacante aprovecha el operador de shell \texttt{;} para separar y encadenar un segundo comando arbitrario al ping legítimo, de la forma \texttt{127.0.0.1; <comando>}.
  \item \textbf{Ejecución en el servidor}: El servidor ejecuta ambos comandos en el contexto del usuario que corre el proceso Apache (\texttt{www-data} o equivalente). En Metasploitable2, este proceso tiene permisos suficientes para leer \texttt{/etc/shadow}.
  \item \textbf{Exfiltración de datos}: La salida de los comandos inyectados se refleja directamente en la respuesta HTML de la aplicación, exponiendo el contenido de los archivos al atacante.
\end{enumerate}

\subsection{Resultado}

Se obtuvieron los archivos \texttt{/etc/passwd} y \texttt{/etc/shadow} del servidor Metasploitable2 directamente desde la interfaz web de DVWA. El archivo \texttt{/etc/passwd} reveló 19 cuentas de usuario del sistema, incluyendo cuentas con shell interactiva (\texttt{msfadmin}, \texttt{user}, \texttt{service}, \texttt{postgres}). El archivo \texttt{/etc/shadow} expuso los hashes de contraseñas de dichas cuentas, información que será utilizada en la Tarea 5 para realizar cracking de contraseñas fuera de línea con John the Ripper.

\section{Tarea 4: Fuerza Bruta a DVWA}

\subsection{Descripción de la vulnerabilidad}

El ataque de fuerza bruta consiste en probar sistemáticamente combinaciones de credenciales hasta encontrar las válidas. DVWA expone este tipo de vulnerabilidad a través de su módulo \textit{Brute Force}, que carece de mecanismos de protección como bloqueo de cuenta tras intentos fallidos, CAPTCHA o limitación de tasa (\textit{rate limiting}). Adicionalmente, el formulario de autenticación principal de DVWA (\texttt{/dvwa/login.php}) presenta la misma debilidad: responde con un mensaje predecible (\texttt{Login failed}) ante credenciales incorrectas, lo que permite automatizar el ataque de forma trivial.

\subsection{Herramientas utilizadas}

\begin{itemize}
    \item \textbf{Burp Suite Community Edition} — proxy HTTP utilizado para interceptar y analizar las peticiones del formulario de autenticación de DVWA, identificar los parámetros de la solicitud e configurar el ataque de diccionario mediante el módulo \textit{Intruder}.
    \item \textbf{Hydra v9.5} — herramienta de fuerza bruta en red que automatiza el envío de credenciales contra el formulario de login HTTP. Se utilizó el módulo \texttt{http-post-form} con el diccionario \texttt{rockyou.txt}.
    \item \textbf{rockyou.txt} — diccionario de contraseñas con más de 14 millones de entradas, ubicado en \texttt{/usr/share/wordlists/rockyou.txt} en Kali Linux.
\end{itemize}

\subsection{Funcionamiento del ataque}

El ataque de fuerza bruta automatizado opera en dos etapas complementarias. En la primera, Burp Suite actúa como proxy interceptando la petición HTTP legítima, lo que permite identificar la estructura exacta del formulario (parámetros, método, URL). El módulo \textit{Intruder} reutiliza esa petición real sustituyendo el valor del campo \texttt{password} por cada entrada del diccionario en sucesivos intentos.

En la segunda etapa, Hydra envía peticiones POST a \texttt{/dvwa/login.php} reemplazando las marcas \texttt{\^{}USER\^{}} y \texttt{\^{}PASS\^{}} por el usuario objetivo y cada contraseña del diccionario \texttt{rockyou.txt}. Para cada respuesta recibida, Hydra verifica si contiene la cadena de fallo \texttt{Login failed}: si está presente, la contraseña es incorrecta; si no está presente, se declara la credencial como válida. Con 16 tareas paralelas, Hydra procesó 14\,344\,399 contraseñas potenciales a velocidad de red, encontrando la correcta en menos de 2 segundos.

La ausencia de controles de seguridad en DVWA (sin bloqueo por intentos fallidos, sin CAPTCHA, sin limitación de tasa) hace que el ataque sea completamente efectivo.

\subsection{Resultado}

Ambas herramientas confirmaron las mismas credenciales del administrador de DVWA:

\begin{itemize}
    \item \textbf{Usuario:} \texttt{admin}
    \item \textbf{Contraseña:} \texttt{password}
\end{itemize}

Hydra encontró la contraseña en el intento número 4 de 14\,344\,399, ya que \texttt{password} es la cuarta entrada del diccionario \texttt{rockyou.txt}. El ataque completó en aproximadamente 2 segundos. Esto demuestra que contraseñas débiles y predecibles, combinadas con la ausencia de mecanismos anti-fuerza-bruta, representan un riesgo crítico de seguridad incluso en entornos web protegidos por autenticación.

\section{Tarea 5: Cracking de Contraseñas con John the Ripper}

\subsection{Descripción de la vulnerabilidad}

El cracking de contraseñas fuera de línea (\textit{offline password cracking}) consiste en obtener los hashes almacenados en el sistema y someterlos a ataques de diccionario o fuerza bruta en local, sin interactuar nuevamente con el sistema víctima. En sistemas GNU/Linux, las contraseñas se almacenan como hashes criptográficos en el archivo \texttt{/etc/shadow}, el cual en condiciones normales solo es legible por \texttt{root}.

Metasploitable2 almacena sus contraseñas con el algoritmo \textbf{md5crypt} (\texttt{\$1\$}), una variante de MD5 diseñada para hashing de contraseñas. Aunque en su momento fue considerado aceptable, md5crypt es actualmente vulnerable a ataques rápidos por diccionario debido a la baja complejidad computacional del algoritmo y a que muchos usuarios eligen contraseñas débiles o predecibles. Una vez que un atacante obtiene el archivo \texttt{/etc/shadow} (como se logró en la Tarea 3), puede realizar el cracking en su propia máquina sin restricciones de tiempo ni límites de intentos.

\subsection{Herramientas utilizadas}

\begin{itemize}
  \item \textbf{unshadow}: Utilidad incluida con John the Ripper que combina los archivos \texttt{/etc/passwd} y \texttt{/etc/shadow} en un único archivo en el formato requerido por John. Esto es necesario porque John necesita tanto el nombre de usuario (en \texttt{passwd}) como el hash de la contraseña (en \texttt{shadow}) para operar correctamente.
  \item \textbf{John the Ripper (Jumbo)}: Herramienta de cracking de contraseñas fuera de línea de código abierto. Implementa múltiples modos de ataque de forma automática y secuencial: modo \textit{Single} (deriva contraseñas candidatas a partir del nombre de usuario y campos GECOS), modo \textit{Wordlist} (ataque de diccionario usando \texttt{/usr/share/john/password.lst}) y modo \textit{Incremental} (fuerza bruta por charset). Detecta automáticamente el tipo de hash y paraleliza el trabajo con OpenMP.
\end{itemize}

\subsection{Pasos realizados}

\subsubsection{Obtención del archivo /etc/shadow}

Dado que el contenido de \texttt{/etc/shadow} no había sido guardado en texto plano durante la Tarea 3, se reutilizó la sesión root obtenida mediante el backdoor vsftpd (Tarea 1) para recuperar el archivo directamente desde la shell interactiva:

\begin{lstlisting}[language=bash, caption={Obtención de /etc/shadow desde la shell root del backdoor vsftpd}]
cat /etc/shadow
\end{lstlisting}

  \centering
  \includegraphics[width=0.92\textwidth]{../img/tarea5/msfconsole_exploit_cat_etcshadow_kali.png}
  \caption{\textit{Contenido de /etc/shadow obtenido desde la shell root del backdoor vsftpd 2.3.4, mostrando los hashes md5crypt de los usuarios del sistema.}}
\end{figure}

\subsubsection{Preparación de los archivos}

En una terminal de Kali Linux se creó el directorio de trabajo \texttt{\textasciitilde{}/tarea5} y se guardaron los contenidos de \texttt{/etc/passwd} y \texttt{/etc/shadow} en los archivos \texttt{passwd.txt} y \texttt{shadow.txt} respectivamente.

\subsubsection{Combinación con unshadow}

Se ejecutó \texttt{unshadow} para generar el archivo \texttt{combined.txt} que contiene, por cada usuario, el nombre de cuenta junto con su hash de contraseña en un único registro:

\begin{lstlisting}[language=bash, caption={Combinación de passwd y shadow con unshadow}]
unshadow passwd.txt shadow.txt > combined.txt
\end{lstlisting}

\subsubsection{Cracking con John the Ripper}

Se ejecutó John the Ripper sobre el archivo combinado. John detectó automáticamente el tipo de hash \texttt{md5crypt} y cargó 7 hashes con 7 saltos distintos, utilizando 4 hilos OpenMP:

\begin{lstlisting}[language=bash, caption={Ejecución de John the Ripper para cracking de hashes md5crypt}]
john combined.txt
\end{lstlisting}

John procedió de forma automática con tres modos en secuencia. En el modo \textit{Single} crackeó inmediatamente cuatro contraseñas cuyo valor coincide con el nombre de usuario (\texttt{user}, \texttt{postgres}, \texttt{msfadmin}, \texttt{service}). En el modo \textit{Wordlist} encontró dos contraseñas adicionales (\texttt{123456789} para \texttt{klog} y \texttt{batman} para \texttt{sys}). El séptimo hash (\texttt{root}) no fue crackeado en el tiempo disponible durante el modo \textit{Incremental}.

\subsection{Funcionamiento del ataque}

El ataque de cracking fuera de línea opera en tres fases:

\begin{enumerate}
  \item \textbf{Preparación}: \texttt{unshadow} fusiona \texttt{/etc/passwd} y \texttt{/etc/shadow} en un único archivo donde cada línea contiene el usuario y su hash. Esto permite a John asociar cada hash con su nombre de cuenta para el modo Single.
  \item \textbf{Ataque por diccionario}: John calcula el hash md5crypt de cada contraseña candidata (nombre de usuario, variantes con mayúsculas/números, y luego entradas del diccionario \texttt{password.lst}) y lo compara contra los hashes almacenados. Cuando hay coincidencia, la contraseña queda revelada en texto plano.
  \item \textbf{Fuerza bruta incremental}: Si el diccionario no agota todos los hashes, John genera combinaciones de caracteres de forma sistemática por charset (\texttt{ASCII}), probando todas las posibilidades de longitud creciente. Este modo puede encontrar contraseñas arbitrarias pero requiere tiempo proporcional al espacio de búsqueda.
\end{enumerate}

La velocidad del ataque depende del algoritmo de hash: md5crypt, al ser relativamente rápido y carecer de trabajo adaptativo (como bcrypt o Argon2), permite procesar en torno a 111\,000 hashes por segundo en el hardware disponible (Apple M4 con emulación ARM), lo que hace viable el cracking de contraseñas débiles en segundos o minutos.

\subsection{Resultado}

John the Ripper crackeó \textbf{6 de los 7 hashes} presentes en el archivo \texttt{/etc/shadow} de Metasploitable2:

\begin{table}[H]
  \centering
  \caption{\textit{Contraseñas crackeadas por John the Ripper en Metasploitable2}}
  \begin{tabular}{@{}lll@{}}
    \toprule
    \textbf{Usuario} & \textbf{Contraseña} & \textbf{Modo de cracking} \\
    \midrule
    user     & user      & Single (nombre de usuario) \\
    postgres & postgres  & Single (nombre de usuario) \\
    msfadmin & msfadmin  & Single (nombre de usuario) \\
    service  & service   & Single (nombre de usuario) \\
    klog     & 123456789 & Wordlist \\
    sys      & batman    & Wordlist \\

\end{table}

Cuatro de las seis contraseñas crackeadas son idénticas al nombre de usuario, lo que evidencia una política de contraseñas nula en Metasploitable2. Las contraseñas \texttt{123456789} y \texttt{batman} fueron encontradas en el diccionario \texttt{password.lst}, confirmando que son contraseñas comunes y predecibles. El hash de \texttt{root} no fue crackeado en el tiempo disponible con el modo incremental, aunque podría serlo con mayor tiempo de cómputo o un diccionario más amplio como \texttt{rockyou.txt}.

\section{Conclusiones}

El desarrollo de esta actividad permitió demostrar de forma práctica cómo una serie de vulnerabilidades conocidas en Metasploitable2 puede ser encadenada para comprometer completamente un sistema. A través de cinco vectores de ataque distintos se evidenció que la seguridad de un entorno no depende de un único control, sino de la suma de todos sus componentes.

La explotación del backdoor en vsftpd 2.3.4 ilustró el riesgo crítico de usar software cuya integridad no ha sido verificada: una sola línea de código malicioso insertada en el repositorio oficial fue suficiente para entregar acceso root sin autenticación. El ataque de SQL Injection sobre DVWA mostró que la concatenación directa de entrada del usuario en consultas SQL sigue siendo una de las vulnerabilidades más prevalentes y devastadoras, permitiendo la extracción completa de la base de datos incluyendo credenciales.

La ejecución de comandos vía DVWA demostró que la falta de sanitización de entrada en funciones de sistema no solo expone datos sensibles, sino que puede comprometer archivos protegidos como \texttt{/etc/shadow}, normalmente accesibles solo por root. El ataque de fuerza bruta con Hydra y Burp Suite evidenció que la ausencia de controles anti-automatización convierte cualquier formulario de autenticación en un blanco trivial para herramientas de diccionario.

Finalmente, el cracking offline con John the Ripper mostró que el almacenamiento de contraseñas con algoritmos débiles como md5crypt, combinado con contraseñas predecibles (o idénticas al nombre de usuario), permite recuperar las credenciales del sistema en segundos. En conjunto, los resultados de esta actividad refuerzan la importancia de aplicar principios de defensa en profundidad: actualizar software, validar entradas, implementar políticas de contraseñas robustas y monitorear el acceso a archivos críticos del sistema.
